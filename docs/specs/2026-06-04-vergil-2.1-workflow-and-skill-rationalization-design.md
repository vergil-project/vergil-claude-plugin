## 1. Context and motivation

Vergil Tooling 2.1 makes two structural changes that ripple through every
skill in this plugin:

1. **A multi-role identity model.** Agents now act under distinct identities —
   `USER`, `AUDIT` — with a reserved `HUMAN` mode and a reserved (unused)
   admin role. Each agent identity is backed by its own GitHub App credential.
2. **A moved agent/human boundary.** The agent no longer drives a PR to
   completion. It prepares a handoff artifact and stops; the **human** opens
   the PR. Work then flows back to the agents for the CI/review loop.

The old `pr-workflow` skill — built on "the agent runs `vrg-submit-pr` end to
end" — is therefore obsolete, and the rest of the collection must be
re-evaluated against the new model. This spec captures the workflow that
replaces it and the resulting skill dispositions.

## 2. The identity model

Three runtime identities (`vergil_tooling.lib.identity_mode.IdentityMode`):

| Mode | Who | Access | Oversight |
| --- | --- | --- | --- |
| `HUMAN` | The Chief Steward (you) | full | n/a — holds all authority |
| `USER` | Code-writing agent (Opus) | write via `vergil-user[bot]` | medium |
| `AUDIT` | Review agent (Sonnet) | least — writes only its feedback artifact + its GitHub review/check, via `vergil-audit[bot]` | least |
| *admin* | reserved | highest | most — **not delegated; no use case yet** |

**The boundary is a credential, not a convention.** `VRG_IDENTITY_MODE` is a
Layer-1 ergonomic only; the real identity boundary is the **GitHub App
credential provisioned to each VM**. This is why the AUDIT identity
structurally has the least access and cannot push code or open PRs — and why
GitHub interactions are legible: `W. Phillip Moore` is the human, `*[bot]` is
an agent. Preserving that legibility is a load-bearing goal of the design.

The reserved **admin role is for an agent acting under the closest human
oversight** (highest privilege ⇒ most oversight). No current use case justifies
delegating it; the human retains all admin authority. The merge override in
§10 is a **human** action, not this role.

## 3. Design principles

These constrain every skill and tool below.

- **Subscription, not API.** Everything runs today through interactive Claude
  Code on a personal subscription — no API automation. Every step must
  therefore be drivable by a human pasting a one-liner.
- **Every step is a future automation seam.** The long-term direction is
  GitHub-triggered automation and, eventually, local LLMs owning the
  mechanical work. Steps are designed as clean seams that can later be
  mechanized.
- **Compact skills (~1–2 pages), one job each.** Historical complexity is
  *dissolved*, not ported.
- **Anything fully deterministic leaves the AI entirely.** If a task can be a
  mechanized utility, it becomes one (see `dependency-update`, §11).
- **AI audit is delta-only.** Whole-tree scanning is the mechanized gates' job
  (Ruff, MyPy, CodeQL, Semgrep, Trivy). The AI reviews only the change.
- **Nothing bypasses the pipeline.** Full gates + audit run every time, even a
  typo fix. The two agents have complementary contexts: USER *solves*, AUDIT
  *holds it accountable*.
- **No honor system.** Controls must be enforced, not requested. A rule the
  agent merely "knows" it should follow is not a control.
- **Defense in depth, human as final gate.** The merge gate is *enforced* (a
  pinned required check, §10) **and** the human watches both agents. Trusting
  the AI means taking responsibility for watching it — the human is the last
  line, not a rubber stamp. The inverse matters: relying *only* on the human
  backstop would be the honor system above, which is exactly why the enforced
  layer must actually hold.
- **Check-and-balance pairs:** brainstorm ↔ pushback · writing-plans ↔
  alignment · **implement ↔ audit**.

## 4. Workflow overview

*(figure 01 — workflow overview)*

Six phases, with the human controlling every boundary crossing:

1. **Design** — brainstorm a spec/plan (large work) or a quick issue (small
   work). Both produce *an issue*, the unit of work.
2. **Implement** + 3. **Local audit** — merged into one parallel pair (§5).
4. **Open PR** — the human runs `vrg-submit-pr` (§9).
5. **Post-PR gates** — both agents loop on the PR; CI + audit gate it (§9–10).
6. **Merge** — the human merges once green and approved.

Executing-plans is a separate wrapper layer that fans issues out as sub-tasks;
**the unit of design here is a single issue.**

## 5. Phases 1–3 — design, issue, implement+local-audit pair

The key structural move: **implement and local audit are one parallel pair,
bounded by a single human round-trip.** "Implement issue #N" is handed to
*both* agents at once. The USER agent implements; the AUDIT agent waits for the
done-signal, then reviews. They communicate through files in a shared
directory.

*(figure 04 — local handshake: worked example + the three endings)*

The collapsed USER-agent task:

1. Implement the issue on the branch.
2. Run all local validation until green.
3. Write the PR handoff file (carrying the commit IDs) — this **is** the
   "done" signal.
4. `vrg-await` the audit feedback; act on the verdict; loop until approved.

The AUDIT-agent task:

1. `vrg-await` the handoff file.
2. Review **only the changed commits**. The audit identity is **read-only by
   discipline, not by sandbox** — it has write access to the shared worktree
   like any process, but is required to touch nothing except its own
   `.vergil/audit-feedback.yml` (and, post-PR, its GitHub review/check). This
   discipline is cooperative; only the GitHub merge gate (§10) is *enforced*.
3. Write a verdict; loop on each new commit until approval.

**Three ways the local loop ends:**

- **Approve** — immediately or after warnings are fixed. The approval lists the
  signed-off **commit IDs** (the audit trail).
- **USER agent gives up** — "can't get it green / don't know what to do." It
  stops and asks the human. Normal, expected.
- **AUDIT declares an ERROR** — not a fixable warning. The audit agent
  **withholds the feedback file**, stops, and alerts the human. The USER agent
  stays safely parked on its await (blocking-forever is the feature). This is a
  *non-human-approved exception*; how approval is recorded is deferred (§13).

The **pre-submit local audit is a soft gate** — the human can override at any
time and run `vrg-submit-pr` regardless. The human is the hard gate until work
reaches GitHub.

## 6. The shared channel and the await tool

The shared directory is **`.vergil/`** at the worktree root — gitignored,
worktree-scoped scratch, cleaned up when the worktree is removed. No
issue-keyed subdirectories are needed (the worktree already isolates parallel
work).

> **Load-bearing assumption — shared filesystem.** The USER and AUDIT agents
> run in separate VMs that **share the host filesystem**: all VMs mount the
> host's `~/dev/projects`, so both see the *same* worktree on disk. The local
> file handshake depends on this. If a VM is ever isolated from that shared
> mount (e.g. by egress/isolation hardening — active work nearby), the
> file-based loop breaks and would need a git-synced or API channel instead.

> **`.vergil/` must be gitignored in every consuming repo.** It is in
> `vergil-tooling`'s ignore but **not yet in this plugin repo** — making this
> uniform is a `repo_init` responsibility.

Two files form the bus:

| File | Direction | Status |
| --- | --- | --- |
| `.vergil/pr-template.yml` | USER → AUDIT & `vrg-submit-pr` | **coded** (tooling 2.1.2) |
| `.vergil/audit-feedback.yml` | AUDIT → USER | **to design** |

`pr-template.yml` is the already-coded contract (`pr_template.py`): required
fields `issue`, `title`, `summary` (optional `linkage`, `notes`); written by
the agent via `write_template()`; read by `vrg-submit-pr` in *template mode*;
deleted by `vrg-submit-pr` after the PR is created (it is ephemeral). The
audit-feedback file is a **sibling** — it must not be folded into
`pr-template.yml`, which is strictly submit-pr input.

### 6.1 `vrg-await` — the wait primitive

One tiny tool, no workflow knowledge; the skills wrap it.

```
vrg-await <path> [--since <sha256>]
```

- No `--since` → block until `<path>` **exists**.
- With `--since` → block until it exists **and** its checksum differs from the
  given value.
- Blocks patiently, indefinitely, by default. On return, prints the current
  `sha256`; the skill threads it back as `--since` on the next round.

### 6.2 The change-detection algorithm (must be rock-solid)

Existence-waiting breaks on iteration: when the audit rejects and the USER
agent **rewrites** the template, the file already exists. So rounds 2+ wait on
a *change*:

- **mtime change** is the cheap wake-up trigger.
- **checksum difference** is the authoritative confirmation (mtime is flaky on
  some filesystems).
- **Writers use temp-file + atomic `rename()`** so a reader never observes a
  half-written file and the mtime/checksum flip atomically.

*(figure 04 traces the full ping-pong, threading checksums `c1 → c2`,
`f1 → f2`.)*

## 7. The audit feedback model

*(figure 02 — audit feedback model: severity × destination)*

Every audit finding carries a **severity** and a **routing**:

- **Severity:** `INFO` / `WARNING` / `ERROR`.
- **Routing — the structural piece:**
  - **→ USER agent** (`audit-feedback.yml`): fix-in-code. Standards violations
    and *removable* suppressions. Soft loop.
  - **→ human** (out-of-band approval): sign-off required.

**Category → behavior:**

- `INFO` / approve — lists signed-off **commit IDs** → ends the loop.
- `WARNING` — the USER agent is expected to fix and re-update → another round.
- `ERROR` — the USER agent can't fix → escalates to the human (see §5's third
  ending).

### 7.1 What the audit actually checks (start simple)

The audit covers the **non-mechanizable judgment** the mechanized gates can't —
crucially, the gates *go green the moment a suppression is added*. Build in this
order:

1. **Coding-standards compliance** — docstrings on *production* code (tests are
   exempt — an accepted blanket exception), naming, structure.
2. **Suppression scrutiny** — detect net-new `# type: ignore`, `# noqa`,
   `# nosec`, or broad `pyproject.toml` ignores. If removable without hurting
   integrity, push back and force the real fix (strong stance on type hints).
3. **Human-approval escalation** — any *newly introduced* suppression requires
   human sign-off. Detection of a net-new suppression is itself mechanizable (a
   diff check) and could become a cheap pre-filter later; the *judgment* and the
   *approval* stay human/AI.

Where sensible, this may piggyback the existing `agentic-review` skill rather
than reinvent it. Keep the initial checks straightforward; the goal is to build
the pipeline, not a maximal review.

## 8. Mechanization and the Diogenes inspiration

*(figure 03 — the mechanized handoff)*

The handoff is mechanized to avoid token-expensive agentic polling, drawing on
the Diogenes MCP workflow-controller pattern:

- **The directory is the state store.** Progress is detected by which files
  exist/changed, not by what the agent claims. `.vergil/` already is this.
- **Deterministic code, not the AI, decides what's next.** Diogenes' killer
  feature is mechanization *between* steps — code that re-verifies and drops
  hallucinated agent output. Our analog: the audit gate and the `vrg-await`
  primitive.

**MVP now → controller later (same pattern, two scales):**

- **MVP:** the blocking-wait tools (`vrg-await`, `vrg-pr-await`) over the
  `.vergil/` files and the PR. Tiny, cheap, ships the loop today.
- **Later:** lift Diogenes' state-machine core + a "what next" MCP server to
  drive the whole thing — the extractable framework and the automation seam
  toward GitHub-triggered and local-LLM execution.

> Diogenes' own server-side execute path is currently a stub (mid-refactor).
> Copy the *pattern* and the ~250-line state-machine core, not that half.

## 9. Phases 4–6 — open PR, post-PR loop, merge

*(figure 05 — post-PR handshake; figure 06 — dual await-loops)*

**Phase 4 — Open PR.** The human runs `vrg-submit-pr` (template mode). It reads
`.vergil/pr-template.yml`, pushes, opens the PR, and deletes the template. On
success it **prints the one-liner to paste** — e.g. `/vergil:pr-watch <PR_URL>`.

**Phase 5 — Post-PR loop.** Opening the PR **auto-triggers** the mechanized CI
gates. The human pastes the same one-liner into **both** agent sessions; the
`pr-watch` skill reads its own identity and runs the matching loop. Both agents
run a `vrg-pr-await` loop on the PR; only the stop conditions differ:

- **USER agent** — wake on any PR change; reconcile **three** sources (CI
  failures + audit review comments + human comments) → patch → commit → push.
  **Stop** when all checks are green and the audit has approved; or give up and
  ask the human.
- **AUDIT agent** — wake on each new commit; re-review the delta; post a GitHub
  review and the `vergil-audit/approved` check. **Continue** while
  approved-but-not-yet-green; **stop** when approved and all checks green;
  **error-abort** by rejecting and requiring the human.

The PR's commit stream is the clock — a push re-triggers CI and the next
review. This is the same handshake as §5, with the PR as a public, archived
bus and a **hard** gate.

**`vrg-pr-await` — settle predicate.** There is no webhook ingress on a laptop,
so the tool **polls the GitHub API**. It blocks until **all named required
checks reach terminal conclusions, OR a new review/commit appears**, then
returns the settled state (checks, reviews, head SHA). The "is it settled?"
logic lives in deterministic code, not agent tokens; poll cadence and
rate-limit handling are `vergil-tooling` spec details.

**Phase 6 — Merge.** The human merges once the PR is green and approved.

## 10. The merge gate (enforcement)

*(figure 07 — merge-gate enforcement, Route B)*

The audit approval must be a *real*, GitHub-enforced control — not an honor
system. Two routes were evaluated:

- **Route A (rejected) — required review via CODEOWNERS.** Pins the requirement
  to a specific reviewer, but **GitHub Apps cannot be CODEOWNERS** (a gap open
  since 2020), forcing the audit identity to become a machine-*user* account.
  This sacrifices the clean App identity.
- **Route B (chosen) — required status check.** The audit App posts a check-run
  `vergil-audit/approved` (success only when it approves); branch protection
  marks it **required**. This is how every CI app already gates merges.

**Route B wins:** both bots stay GitHub Apps (identity model untouched), it
mirrors the local loop exactly (write-`approve` → post-pass), findings still
post as PR comments (the archive), and the audit gate becomes **just another
required CI check** beside CodeQL/Semgrep/Trivy. If the audit bot doesn't post
success, the merge is blocked. **Verified against GitHub docs:** a required
check can be pinned to a specific App, and a same-named check from another app
is rejected ("not set by the expected GitHub App").

> **Security-critical configuration (MUST).** The required check must be
> **pinned to the audit App's identity** — `integration_id` = `vergil-audit[bot]`'s
> app id in a ruleset (or `app_id` in classic branch protection). This is
> **opt-in**: left unpinned ("any source" / `app_id` omitted or `-1`), GitHub
> matches by **context name only**, and `vergil-user[bot]` could post its own
> `vergil-audit/approved` and **forge the gate**. Treat the pin as a security
> value: set it *and verify the applied IaC state* (the Terraform GitHub
> provider has had `integration_id` drift quirks — verify state, not just plan).
> Additionally, do **not** grant `vergil-user` the `checks:write` permission, so
> it cannot create a check-run at all (check-runs are GitHub-App-only).

**Per-commit semantics.** The `vergil-audit/approved` check is bound to a head
SHA, so a new push by the USER agent **invalidates the prior approval** — the
new commit is unapproved until the audit re-posts. This is the *desired*
behavior (it forces re-review of new commits). Enable **dismiss-stale-approvals
/ require approval of the most recent push** so a green-then-amended PR cannot
slip through.

The audit check is **required and non-optional per repo** — bypassing it means
not using the system as designed. Combined with the human watching both agents
(§3), this is **defense in depth**: the enforced check is primary, the human is
the backstop.

**Human override** is a deliberate, logged repo-admin action: leave "Do not
allow bypassing" off (simple), or — preferred on an org repo — a ruleset
**bypass actor** = a one-member team (the human), "for pull requests only," so
overrides stay audit-logged. This is a *human* override, not the reserved admin
agent role.

> Grounded in current GitHub docs (available-rules-for-rulesets, branch-protection
> REST API, troubleshooting-required-status-checks, and the 2021-11-30 "ensure
> required status checks provided by the intended app" changelog). Re-verify if
> GitHub ships app-as-CODEOWNER, which would reopen Route A.

## 11. Skill rationalization (issue #427)

*(figure 08 — skill disposition map)*

From the seven existing skills: **5 keep · 2 retire · 3 new.**

### Existing skills

| Skill | Disposition | Rationale |
| --- | --- | --- |
| `pr-workflow` | **retire → split** | Built on the dead "agent submits PR" model. Dissolved into `implement` / `pr-watch` + the human running `vrg-submit-pr`. The heart of the change. |
| `dependency-update` | **retire → mechanize** | 100% deterministic (despite language-support variety) ⇒ leaves the AI umbrella entirely; becomes a mechanized `vergil-tooling` utility. |
| `handoff` | **keep** | Not retired. Its session-continuity recovery covers a real case the new design doesn't: a laptop reboot mid-task with unsaved, un-externalized agent state/decisions. It sits mostly unused, costs nothing to keep, and is the **interim recovery net** while `implement`/`audit` resumability stays deferred (§13). Ensure its "handoff" naming doesn't collide conceptually with the PR `pr-template` handoff. |
| `deprecation-triage` | **keep · verify** | Confirm the acting identity (USER/AUDIT) may create issues; if not, the skill hands issue creation to the human. Otherwise unaffected. |
| `summarize` | **keep** | No boundary impact. |
| `memory-audit` | **keep** | Human-approval gated already; role check only. |
| `memory-init` | **keep** | Unaffected. |

### New skills — each a one-liner the human pastes

| Skill | Identity | Role |
| --- | --- | --- |
| `/vergil:implement <issue>` | USER · local loop | Implement → validate-until-green → write `pr-template.yml` (with commit IDs) → `vrg-await` feedback → fix/loop, escalate ERROR, or done. Resumability from durable state is a goal (§13); `handoff` is the net meanwhile. |
| `/vergil:audit <issue>` | AUDIT · local loop | `vrg-await` template → review the delta (read-only by discipline) → write 3-category feedback / loop; ERROR ⇒ withhold + alert. |
| `/vergil:pr-watch <PR_URL>` | identity-keyed · post-PR | Emitted by `vrg-submit-pr`. USER → monitor & reconcile CI+audit+human → push. AUDIT → re-review & post `vergil-audit/approved`. |

> Open refinement: whether `implement` and the USER half of `pr-watch` (and
> `audit` and the AUDIT half) are separate skills or one identity-keyed skill
> spanning both phases. Resolved during implementation; the disposition above
> is the working split.

## 12. Cross-repo work breakdown

This design spans both repos of the two-repo model. **#427 (this plugin) covers
the skills**; the tooling needs companion issues in **`vergil-tooling`**.

> **Deployment gating.** The plugin skills can be *authored* now but are **not
> deployable until `vergil-tooling` 2.1 ships** (the tools below land and repos
> upgrade to 2.1). The two repos proceed in **parallel**; interim friction
> (agents reaching for the removed `pr-workflow`) is handled manually. Some
> environments still run tooling 2.0.80 (no template mode), so the new skills
> cannot be smoke-tested until 2.1 is installed.

- **NEW** `vrg-await` — blocking file waiter (temp+rename safe, `--since`).
- **NEW** `vrg-pr-await` — blocking PR-state waiter with the §9 settle predicate.
- **EXISTS** `vrg-submit-pr` template mode (2.1.2) — **add** one-liner emission
  on success; optional cross-check of `audit-feedback.yml` before submit.
- **NEW** audit posts the `vergil-audit/approved` check-run (App needs
  `checks:write`; `vergil-user` must **not** have it).
- **NEW** mechanized `dependency-update` utility (absorbs the retired skill).
- **CONFIG** branch-protection / ruleset: required checks (incl.
  `vergil-audit/approved` **pinned to the audit App id**, §10) + human bypass +
  dismiss-stale-approvals.

## 13. Deferred and open items

- **Recording a non-human-approved exception.** When the audit declares ERROR
  (a new, unapproved suppression), *how the human's approval is recorded /
  annotated* is undefined. Deferred deliberately.
- **The MCP workflow controller.** The full Diogenes-style state-machine + "what
  next" server is the post-MVP target, not this iteration.
- **Resumability of `implement`/`audit`.** Making the new skills resume from
  durable state (issue · commits · `.vergil/`) is a goal but **deferred** — and
  **not a blocker**, because the retained `handoff` skill (§11) is the interim
  recovery net.
- **Runaway-loop visibility.** No hard cap on the implement↔audit fight; the
  human watches (both sessions visible) and ERROR escalates. A round counter in
  the feedback file would make a spiral *visible* — optional.
- **`audit-feedback.yml` format.** The file's exact schema (severity, routing,
  per-commit verdicts, approved commit IDs) is to be specified during
  implementation.
- **Physical cross-repo split.** Migrating §§1–10 into a `vergil-tooling` design
  spec (per the scope note) is a tracked follow-up needing its own issue +
  worktree.
- **Skill-split refinement** (§11) — `implement`/`pr-watch` and `audit`/`pr-watch`
  as separate vs unified identity-keyed skills.
