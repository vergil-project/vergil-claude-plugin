The vergil 2.1 rationalization
(`docs/specs/2026-06-04-vergil-2.1-workflow-and-skill-rationalization-design.md`)
moved the agent/human boundary and renamed or removed several tools.
The plugin's hooks and docs still encode the pre-2.1 workflow. This
spec inventories every disagreement and defines the reconciled target
state. First instances of the drift class were fixed under #428
(`enforce-host-container-split` matching the renamed `vrg-docker-run`;
dead `HOST_TOOLS` entries).

## 1. Ground truth (decisions confirmed 2026-06-05)

These are the workflow facts the hooks must encode. Confirmed with the
human during this design's brainstorm:

1. **`vrg-finalize-pr` is a human-only tool** — never run by USER or
   AUDIT agents.
2. **Merges happen in the GitHub UI today.** Target state: the human
   runs `vrg-finalize-pr <pr>` and it performs the merge from the CLI
   as well (missing feature — companion tooling issue, appendix A.1).
3. **Agents structurally cannot create or merge PRs** — the per-VM
   GitHub App credentials do not allow it. Hooks that police this are
   ergonomic fast-fails on top of a hard credential gate.
4. **Issues are closed explicitly by the human after finalization** —
   never by auto-close keywords (the Ref-only linkage policy stands).
   Closing is deliberately manual today: agents have closed issues
   incorrectly, so no automation home exists yet (see §6 for the
   possible future close-analysis agent).
5. **`vrg-finalize-repo` no longer exists.** `vrg-check-pr-merge` was
   never shipped.

## 2. Soft-gate doctrine

Every guard inside the VM is a **soft gate**, accepted as such: it
steers a correctly behaving agent toward the right default. A
misbehaving agent can unset `VRG_IDENTITY_MODE`, rewrite its
environment, or route around any hook — and nothing in the VM can
prevent that. Hard enforcement lives outside the VM: per-identity
GitHub App credentials, the `vergil-audit/approved` required check
pinned to the audit App, and the VM sandbox itself. Consequence for
design: hooks stay simple. No config-file identity resolution, no
attempt to make in-VM enforcement tamper-proof.

Corollary — **hooks restrict the agent regardless of identity by
default.** Identity-keyed behavior (the write-guard, §3.3) is the
per-hook-justified exception, not the rule. `VRG_IDENTITY_MODE=human`
is set nowhere in practice: Claude running with human credentials is
a guards-down scenario being eliminated universally. The direction of
travel is the opposite — agents hand commands to the human to run,
never the reverse.

### 3.3 New capability (#442): `guard-audit-writes.sh`

PreToolUse, matcher `Write|Edit|NotebookEdit`.

- **Identity:** `VRG_IDENTITY_MODE` env var. Not `audit` (or unset) →
  exit 0. Gated on `is_managed_repo` like every other hook. The env
  var is the 2.1 spec's "Layer-1 ergonomic" — per §2 this is a soft
  gate and that is accepted.
- **Allowlist** (identity = audit; deny everything else):
  - `.vergil/audit-*` — the audit's own artifacts. Deliberately **not**
    all of `.vergil/`: `pr-template.yml` is the USER agent's artifact
    and the human's `vrg-submit-pr` input; AUDIT overwriting it would
    corrupt the handshake.
  - `build/**` — audit scratch space (e.g. side-channel notes,
    generated review aids).
- **Path handling:** lexical normalization (`..` resolution, absolute
  vs relative against cwd). No `realpath` dependency — host hooks must
  run on macOS bash 3.2 (the same constraint behind
  `block-associative-arrays`). **Deny conditions** (identity = audit):
  the tool payload's path field is missing or empty, or the
  normalized path escapes the worktree root (leading `..` after
  normalization, or an absolute path outside the worktree).
  Fail-closed inside the audit identity, fail-open outside it.
- **Bash writes: out of scope.** Detecting file writes in arbitrary
  Bash commands (`>`, `tee`, `sed -i`, `cp` …) is heuristic and
  false-positive-prone. Documented gap; covered by discipline and
  credentials. Revisit only if drift is observed (§6).

**Testing:** table-driven plain-bash test script
`hooks/tests/guard-audit-writes.test.sh` — feeds the hook JSON
payloads across an identity × path × tool matrix and asserts each
decision. Bash-3.2-compatible. Runs directly; wiring into
`vrg-validate` is deferred (§6).

## 6. Deferred

- **Bash-write coverage** for the write-guard — only if observed drift
  justifies the false-positive risk.
- **Approve-verdict reminder** — a PostToolUse successor to
  `remind-finalize` keyed to something agents actually run (e.g.
  `vrg-await` returning an approve verdict), reminding the agent to
  hand the human the `vrg-submit-pr` one-liner. Only if its absence is
  felt in practice.
- **Wiring `hooks/tests/` into `vrg-validate`** — needs a shell-test
  step in the tooling validation pipeline.
- **Automated issue closing** — closing is manual today because agents
  have closed issues incorrectly. Likely shape: a dedicated agent that
  analyzes whether the work is truly complete and closes (or
  recommends closing) accordingly. Not designed here.

### A.1 `vrg-finalize-pr`: merge the PR from the CLI

> **Title:** vrg-finalize-pr: perform the merge itself when given a PR
>
> `vrg-finalize-pr <pr>` currently expects the merge to have happened
> in the GitHub UI. Target state per the 2.1 workflow: the human's
> entire Phase-6 action is one CLI command — merge (right strategy,
> provenance check), switch to the target branch, pull, prune. The
> GitHub UI should only be needed when something is genuinely
> exceptional. Out of scope: issue closing stays a manual human
> action (no reliable automation home yet).

### A.2 `vrg-check-pr-merge` disposition

> **Title:** vrg-check-pr-merge was never shipped — ship it or delete
> the references
>
> `vergil-claude-plugin`'s `block-agent-merge` hook delegated to
> `vrg-check-pr-merge`, which has no entry point in 2.1 — the hook
> fail-closed on exit 127 with a "tool failed" message. The plugin is
> removing the dependency (the hook becomes an unconditional deny
> since agents have no merge path under 2.1). Decide whether the tool
> has any remaining use case; if not, remove its references from
> tooling plans/docs.

### A.3 `repo_init` gitignore baseline has drifted from the 2.1 spec (critical)

> **Title:** repo_init: gitignore baseline missing .vergil/, build/,
> .superpowers/; tooling's own .gitignore missing `*~`
>
> The 2.1 workflow spec assigns ".vergil/ gitignored in every
> consuming repo" to `repo_init`, but `render_gitignore()` does not
> include it — the mechanism designated to enforce the convention
> doesn't know about it. A 2026-06-05 three-way comparison
> (vergil-claude-plugin `.gitignore` · vergil-tooling `.gitignore` ·
> `render_gitignore()`) found drift in both directions:
>
> 1. Add to `render_gitignore()`: `.vergil/`, `build/`,
>    `.superpowers/`.
> 2. Add to vergil-tooling's own `.gitignore`: `*~` (emacs backups —
>    present in the baseline, absent in the flagship repo).
>
> Consider whether the baseline and the flagship repo's file should be
> generated from one source so they cannot drift again.

### A.4 `vrg-git branch -D`: "upstream gone" gate can never pass for worktree-convention branches

> **Title:** vrg-git: branch -D denied forever for branches tracking
> origin/develop
>
> `vrg-git branch -D` denies deletion unless the branch's upstream is
> gone. But the worktree convention's canonical invocation
> (`git worktree add -b <branch> origin/develop`) sets upstream to
> `origin/develop` — which is never gone. Result: every locally
> abandoned or squash-merged branch created this way is permanently
> undeletable through the wrapper (observed with
> `chore/417-bump-2.0.17`; the human had to fall back to raw
> `git branch -D`). The gate should treat "upstream is a protected
> integration branch" (or "no same-named remote branch exists") as
> deletable, or offer an explicit safe path for squash-merged
> branches.

### A.5 `local.claude_settings` audit check: verify content, not just existence

> **Title:** vrg-github-repo-config: claude_settings check should
> detect plugin-manager clobbering
>
> On 2026-06-04 Claude Code's plugin manager rewrote a checked-in
> `.claude/settings.json` in place — emptying `enabledPlugins` and
> `extraKnownMarketplaces` — during a routine plugin refresh
> (adjacent to Claude Code bug #51118). The repo-config audit's
> `local.claude_settings` check only verifies the file exists, so
> this drift is mechanically invisible. Extend the check to verify
> the marketplace/plugin keys match the canonical template, so the
> config-compliance gate catches it at the next PR.
